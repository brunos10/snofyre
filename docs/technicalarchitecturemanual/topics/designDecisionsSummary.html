Snofyre Technical Architecture Manual - Draft

Design Decisions Summary

This section summarises the decisions relevant to the Information Model and the backend Relational Database schemas that are implemented in the Snofyre. It also offers a summary of the implications of the decisions on the querying capabilities of Snofyre.

Information Model Decisions

Snofyre uses a propietary information model to persist and retrieve objects (especially when simulated data is being generated). Since the main focus of Snofyre is the Terminology Model rather than the Information Model, the information model used in Snofyre is restricted using the following design principles.
  • The Information Model will only include entities that are essential for clinical records with a primary focus on the terminology model.
  • The attributes declared on the Information Model classes will be restricted to those defined by the scope of the project. The range and scope of attributes used might change to accomodate any change in the scope of the project in the future.
  • Every clinical entity in the Information Model will encapsulate a SNOMED CT concept (or clinical expression) where appropriate.
  • No conscious effort was made to align the different entities in the Information Model or the record structure itself with any known or published standards.

For a more detailed disccussion on this topic, the reader is referred to the topic Information Model.

Database Schema Decisions

In view of the features of the object model - relational model mapping strategies and the scope of the project, the following principles were adopted for deciding the flavour of database schema to be used in Snofyre.
  • It is assumed that querying in the context of secondary uses of clinical data is performed over database tables that are completely denormalised.
  • For the purpose of demonstrating reporting capabilities, it is assumed that it is easier to create denormalised tables from normalised tables than the other way around.
  • It is assumed that denormalised table schemas will be easier to create in any ETL (Extraction, Transformation and Load) procedures that might need to be carried out if Snofyre were to be demonstrated across to a different database.

For a more detailed disccussion on this topic, the reader is referred to the topic Database Schema Designs

Table Creation Strategy Decisions

Since a Query Statement is often composed of more than one sub queries nested within it, Snofyre often needs to compute the result set that satisfy the criteria each of the sub queries before it can compute the result set that satisfies all the criteria encapsulated within the Query Statement. During execution, the result set for each of the sub queries is often calculated and stored as either a Table, Temporary Table or View.
Note: Table Creation Strategy might have an impact on the performance of a query of the Snofyre especially the time required to compute the result set of a given query. In general this is dependant on the way in which indices are created and handled within tables, temporary tables and views within a particular database engine implementation.

Snofyre does not make any assumptions about the most suitable table creation strategy. Instead, the user is expected to choose the most appropriate table creation strategy for their favourite database engine.

For a more detailed disccussion on this topic, the reader is referred to the topic Table Creation Strategy.

Database Dialect

There are many different implementations of the relational database management engines and often every implementation has a particular syntax/flavour of SQL (Structure Query Language) that is used to communicate with the database engine. Snofyre uses dialects to allow the implementer to use their favourite database backend to store test data and plug it into Snofyre.
Note: Snofyre provides out of the box support for only MySQL dialect. Snofyre developers are expected to implement a Database Dialect for their favourite database engine. Please refer to the source code of the MySQLReportingQueryExecutionEngine class on how to implement this.

For a more detailed disccussion on this topic, the reader is referred to the topic Database Dialect.
