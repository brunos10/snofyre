Snofyre Application Framework Overview

This section provides an overview of the various third party and native frameworks used by Snofyre. While the following sections describe the individual frameworks in greater detail, this section describes how these frameowrks are related to each other. The modular, service oriented architecture of Snofyre allows for modules to be linked, which can be visualised as a stack of modules built on top of each other, as shown in... insert figure here

Figure 1. Overview of frameworks used in Snofyre
At the bottom of the stack is the modularity framework which allows Snofyre's various components to interact with each other seamlessly. Snofyre uses Dependency Injection as a means of wiring together its various components in an unintrusive fashion. Snofyre implements a Docking Framework that uses the underlying modularity to allow various user interface components to dynamically respond to interact with Snofyre's services. Snofyre's services are organised as modules which fall into the following broad categories:
  • SNOMED CT Services
  • Search Services
  • Query Services
These services are described in detail later on in Core Services Overview. The user interface components that a Snofyre user interacts with are standard paneles, buttons, tables etc which make appropriate calls to the underlying services to deliver application functionality.

While it is important to understand how all these application frameworks interact with each, in this guide we treat the Modularity and Dependency Injection framework as Low level application frameworks. On the other hand any frameworks that are more specific to Snofyre are treated as High Level application frameworks. We believe that the Low level frameworks are generic enough that the reader can refer to already published resources for additonal details. On the other hand the High level frameworks are specific to Snofyre and are important for understanding how Snofyre's various components interact.
